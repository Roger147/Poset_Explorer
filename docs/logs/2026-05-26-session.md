# Session Log - 2026-05-26

## Main Decisions

- Begin moving expensive structural computation toward a Rust backend, starting
  with transitive closure because many analyzer operations depend on
  comparability.
- Keep Python labels as the public API and introduce canonical integer indices
  as the Rust-facing representation.
- Add `element_to_index` and `index_to_element` to `Poset` so arbitrary element
  labels can still be used safely while Rust receives compact indexed data.
- Put the Rust boundary behind a Python adapter module, `closure.py`, so the
  analyzer does not import the compiled extension directly.
- Keep a pure-Python closure fallback so the project remains usable from a
  source checkout even when the Rust extension has not been built.
- Use Maturin and PyO3 for packaging so users can install the hidden Rust
  extension through normal Python packaging workflows.
- Convert the Rust closure implementation from set-like collection logic to
  bitset rows keyed by element indices.
- Defer deeper Rust-backed optimizations until the analyzer can keep and reuse
  indexed closure data directly instead of immediately mapping everything back
  to label sets.

## Concepts Clarified

- Rust should operate on integer indices, not arbitrary Python labels.
- Python remains responsible for preserving element labels and public API
  behavior.
- The current speedup applies most directly to building transitive closure:
  Rust now performs successor accumulation with `u64` bit operations instead of
  Python set updates over strings.
- Downstream operations benefit only partially for now because the analyzer
  still converts the indexed Rust result back into `dict[str, set[str]]`.
- The next major performance step should be keeping indexed closure data on
  the analyzer and using it directly for comparability queries, intervals,
  zeta summaries, Mobius support, width, and weighted-width constraints.
- PyO3 extension-module code is awkward to unit-test directly through ordinary
  Cargo test linking, so the Rust algorithm should live in plain Rust modules
  with a thin PyO3 wrapper.

## Future Rust-Backed Plans

- Cache indexed closure or bitset closure on `PosetAnalyzer` instead of only
  caching label-based successor sets.
- Rewrite `is_less_equal()` to use indexed bitset membership internally.
- Rewrite `comparable_successors()` and `comparability_edges()` to read from
  indexed closure and map labels only at the return boundary.
- Rewrite interval construction to use closure bitsets or indexed membership
  checks before mapping interval elements back to labels.
- Move zeta matrix and zeta summary computation closer to the indexed closure
  representation.
- Move Mobius matrix computation to Rust after closure/zeta indexing is stable.
- Consider Rust-backed linear-extension counting with bitmask states and
  predecessor masks.
- Consider Rust-backed order-ideal generation only after deciding how much of
  the ideal lattice should be returned as compact encoded state versus Python
  `Ideal` objects.
- Consider Rust-backed width and weighted-width algorithms after comparability
  data stays indexed long enough to avoid unnecessary label/set conversion.

## Future Interval-Customization Plans

- Design a parameterized interval-customization layer rather than adding many
  one-off weighted interval methods.
- Keep the existing closed and open element-weighted interval helpers simple
  until richer interval semantics are ready.
- Support well-known model families through explicit interval semantics, such
  as:
  - probabilistic propagation models,
  - workflow and scheduling models,
  - fluid-flow or capacity-flow interpretations,
  - cover-edge accumulation models,
  - comparability-pair or source-target kernel models.
- Keep model-specific interval behavior separate from canonical structural
  poset operations so `families.py` remains mathematically clean.
- Let future interval-native weighting handle skip-style chain interpretations
  and other semantics that should not be hidden inside `max_chain_score()`.
