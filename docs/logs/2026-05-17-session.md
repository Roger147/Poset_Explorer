## Concepts Clarified

- A stale `Project Structure` section in the README can be worse than omitting
  it, especially while the project is still changing quickly.
- The README should foreground project purpose, conceptual ownership, current
  capabilities, testing, and documentation pointers.
- Large analyzer outputs, such as Mobius matrices, should stay explicitly
  requested rather than being included in ordinary summaries or README examples.
- A weighted poset should share the underlying `Poset` by reference and store
  only weight metadata.
- Reweighting should return a new wrapper over the same `Poset` instead of
  mutating the existing wrapper by default.
- Element weights are keyed by element labels, while edge weights are keyed by
  direct stored cover/dependency relation tuples.
- Edge weights should initially apply to stored Hasse-cover relations, not all
  transitive comparability pairs.
- Random, learned, empirical, or sparse weights all fit naturally as long as
  dictionary keys are valid.
- Weight vectors are ordered numerical views of weight dictionaries, using the
  poset's canonical order for elements and canonical cover-edge order for
  edges.
