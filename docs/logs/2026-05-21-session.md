# Session Log - 2026-05-21

## Main Decisions

- Treat weighted width as an element-weighted invariant only: it measures the
  maximum total element weight of an antichain.
- Do not include edge weights in weighted width for now. Edge-weighted
  "width" needs a separate definition before it should become an API.
- Reserve measurement-style names for nonnegative quantities:
  `weighted_width()`, `max_chain_weight()`, and `ideal_weight()` reject
  negative weights.
- Add separate score-style APIs for signed optimization or signed totals:
  `max_antichain_score()`, `max_chain_score()`, and `ideal_score()`.
- Keep `mode="both"` for chain weights only when element and edge weights are
  measured in compatible units.
- Start notebook-based showcases as a friendlier way to demonstrate library
  behavior outside the test suite.

## Concepts Clarified

- Weighted width is the weighted analogue of ordinary width only when weights
  behave like nonnegative size, mass, cost, duration, or capacity.
- Negative weights turn a width or height measurement into an optimization
  score, so signed behavior should be exposed through score-named methods
  rather than hidden behind an option flag.
- Antichain score allows the empty antichain, so an all-negative antichain
  problem can return `0`.
- Chain score keeps the existing nonempty-chain convention on nonempty posets,
  matching the chain-weight behavior.
- Ideal scores can naturally be signed totals; ideal weights remain
  measurement-style nonnegative totals.
- Weighted width must use transitive comparability, matching the ordinary
  width implementation, not just stored Hasse cover edges.
