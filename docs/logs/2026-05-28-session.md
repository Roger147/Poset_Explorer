# Session Log - 2026-05-28

## Main Decisions

- Add additional safety around linear-extension counting rather than relying
  only on the element-count guard.
- Treat antichains as a safe fast-return case for linear extensions because
  their count is exactly `n!`.
- Keep the linear-extension Rust backend protected by a state-count limit even
  when callers override the Python element-count guard.
- Move weighted width / maximum antichain score into Rust before starting
  interval-native API design.
- Keep weighted width conceptually separate from structural width:
  structural width is unweighted matching, while weighted width is a weighted
  max-antichain/min-cut problem.
- Start the interval-native layer with only minimal Python scaffolding and no
  Rust acceleration yet.
- Keep canonical `PosetAnalyzer.interval()` and `interval_summary()` as
  structural poset operations; interval interpretation belongs in a separate
  interval-native module.

## Concepts Clarified

- Linear-extension guards now have two different roles:
  - `max_elements` protects callers before entering the exponential backend,
  - `max_states` protects memoization growth after the backend starts.
- Empty posets, chains, and antichains are now all fast-return cases before
  the linear-extension element guard.
- Antichains are safe to identify through the already Rust-backed structural
  width calculation.
- Weighted width is equivalent to maximum element-weight antichain, while
  signed antichain scoring allows negative weights by selecting the empty
  antichain if needed.
- Weighted width and signed antichain score can share the same Rust backend;
  the only difference is that `weighted_width()` rejects negative element
  weights first.
- The interval-native layer should not replace canonical structural interval
  methods.
- `PosetAnalyzer.interval_summary()` remains a canonical closed-interval
  structural summary over the poset closure.
- `WeightedIntervalAnalyzer` is the first home for interval interpretations,
  but richer probabilistic, workflow, flow, cover-edge, or kernel-style models
  should wait until example models are prepared.
- Rust acceleration for interval-native models is deferred until the semantic
  surface is clearer.
