# Session Log - 2026-05-27

## Main Decisions

- Document repository setup in the README so contributors can create the
  virtual environment, install dependencies, and build the hidden Rust
  extension consistently.
- Keep Rust-backed computations behind `closure.py` adapter functions rather
  than importing `_poset_explorer_rust` directly from analyzer code.
- Cache indexed analyzer data where useful, while preserving the public
  label-based Python API.
- Move compact structural summaries to Rust before object-constructing APIs.
- Keep `get_lattice_layers()` as a Python object-construction path for now,
  while moving `lattice_layer_sizes()` and summary-style ideal counts through
  Rust.
- Keep canonical `interval(x, y)` structural and label-returning; richer
  interval semantics should wait for a future interval-native configuration
  layer.
- Use `f64` for Rust-backed zeta transform totals, accepting float outputs as
  the intended transform behavior.
- Add a default element-count guard for linear-extension counting, with empty
  posets and chains returning before the guard.
- Leave weighted width and weighted lattice-layer summaries for later design
  because they involve weighted optimization and object/materialization
  semantics beyond the current structural backend.

## Concepts Clarified

- `zeta_matrix()` still constructs the public label-keyed Python dictionary,
  but its comparability checks use indexed closure data.
- `zeta_summary()` is separate from `zeta_matrix()` and uses compact Rust
  summary data directly.
- Weighted zeta transform remains a thin weighted wrapper over the base
  analyzer's closed zeta transform, so it benefits from the Rust-backed
  transform path automatically.
- Weighted Mobius inversion remains delegated to the base analyzer because the
  relation is still ordinary Mobius on the underlying unweighted poset.
- Structural width and weighted width are distinct problems:
  - structural width now uses Rust-backed matching,
  - weighted width still uses Python max-flow over weighted antichain
    capacities.
- Linear-extension counting maps cleanly to Rust bitmasks and memoization, but
  it remains exponential; Rust improves execution but does not remove the
  combinatorial blowup.
- Antichains remain guarded like general non-chain posets for now. Only empty
  posets and chains receive fast-return treatment before the linear-extension
  guard.
- Weighted lattice-layer summaries should probably not be Rustified by passing
  Python `Ideal` objects around. A later compact weighted summary backend
  should compute directly over indexed ideal bitsets if those semantics are
  needed.
