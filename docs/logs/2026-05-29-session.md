# Session Log - 2026-05-29

## Main Decisions

- Treat the "Gemini virtual matrix idea" transcript as design inspiration for
  the interval-native layer, not as an exact implementation plan.
- Frame interval-native analysis as a lazy virtual interval valuation matrix:
  each coordinate `(x, y)` corresponds to the structural interval `[x, y]`
  when `x <= y`, and to a model-specific zero or empty value otherwise.
- Keep the analyzer responsible for interval coordinates and structural
  extraction, while future kernels or models define how each interval cell is
  valued.
- Avoid dense interval-matrix materialization by default; matrix-like access
  should behave as a lazy view.
- Keep the current `WeightedIntervalAnalyzer` scaffold small until concrete
  example models are prepared.
- Align signed chain and antichain scoring semantics: both are global
  optimizations and may select the empty object.
- Leave negative scores meaningful in contexts where the substructure is
  explicitly chosen, such as ideals and intervals.

## Concepts Clarified

- A virtual interval matrix should not replace canonical structural interval
  operations.
- `PosetAnalyzer.interval(x, y)` remains the canonical structural closed
  interval.
- `PosetAnalyzer.interval_summary()` remains the canonical closed-interval
  structural summary over the poset closure.
- A future interval-native analyzer can expose matrix-like access, such as
  `analyzer[x, y]`, while delegating valuation to a kernel/model.
- The first scalar interval model is element-sum over configured endpoints,
  but future models could represent probabilistic, workflow, flow,
  cover-edge, or kernel-style interpretations.
- `max_chain_score()` and `max_antichain_score()` are now both global
  signed optimizations, so both can return 0 by choosing the empty chain or
  antichain.
- `ideal_score()` and interval valuations can still be negative because they
  score specific selected substructures rather than choosing a globally
  optimal empty object.
